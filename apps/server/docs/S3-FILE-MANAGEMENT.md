# S3/MinIO 文件管理指南

## 问题说明

之前发现MinIO桶中的视频文件被意外删除的问题。经过排查：

1. **代码中没有自动删除文件的逻辑** - 所有删除操作只删除数据库记录，不删除S3文件
2. **MinIO没有配置生命周期策略** - 文件不会自动过期删除
3. **文件可能是手动删除的** - 需要注意不要在MinIO控制台手动删除文件

## 文件保护机制

### 1. 删除数据库记录时不删除文件

当你在后台管理系统删除分集或短剧时：
- ✅ 只删除数据库记录
- ✅ S3/MinIO中的文件保留
- ✅ 防止误删除仍在使用的文件（多个分集可能共享同一个视频）

### 2. 定期清理孤立文件

使用专门的清理脚本来删除不再使用的文件：

```bash
# 预览模式（不实际删除，只显示哪些文件会被删除）
cd apps/server
node lib/s3-safe-delete.js

# 实际删除孤立文件
node lib/s3-safe-delete.js --delete
```

### 3. 检查文件状态

```bash
# 检查MinIO中的文件与数据库记录是否匹配
node scripts/check-s3-files.js

# 检查MinIO桶的配置
node scripts/check-bucket-config.js
```

## 最佳实践

### ✅ 推荐做法

1. **不要在MinIO控制台手动删除文件** - 这会导致数据库记录与实际文件不匹配
2. **使用后台管理系统删除** - 只删除数据库记录，文件保留
3. **定期运行清理脚本** - 每周或每月清理一次孤立文件
4. **删除前先预览** - 使用预览模式查看哪些文件会被删除

### ❌ 避免做法

1. ❌ 不要直接在MinIO控制台删除文件
2. ❌ 不要修改代码添加自动删除文件的逻辑
3. ❌ 不要在没有备份的情况下批量删除

## 文件共享说明

**重要：多个分集可以共享同一个视频文件**

这是正常的设计，因为：
- 上传相同的视频文件时，系统会检测MD5哈希值
- 如果文件已存在，会复用现有文件的URL
- 这样可以节省存储空间

**因此，删除一个分集时不能删除其视频文件，因为其他分集可能还在使用！**

如果发现文件丢失：

1. 重新上传视频文件
2. 在后台管理系统编辑对应的分集
3. 重新选择上传的视频文件
4. 保存更新

## 技术细节

### 文件存储路径

```
fastshort/
├── video/
│   ├── 20260224/
│   │   └── {md5hash}.mp4
│   ├── 20260225/
│   │   └── {md5hash}.mp4
│   └── 20260226/
│       └── {md5hash}.mp4
└── avatar/
    └── {date}/
        └── {md5hash}.{ext}
```

### 文件URL格式

```
http://localhost:9000/fastshort/video/20260226/{md5hash}.mp4
```

### 数据库字段

```javascript
{
  video: ['http://localhost:9000/fastshort/video/...'],  // 数组格式
  cover: ['http://localhost:9000/fastshort/video/....png']
}
```

## 监控建议


如果发现不匹配，及时处理。
